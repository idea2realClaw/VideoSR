## 第二轮：图片超分进度改为 Tile 计数（commit 0bd7c8b）

### 需求

### 改动
- **server.py**：`upsample()` 新增可选参数 `tile_progress_callback(processed, total)`；每处理一个 Tile 后 `tile_count += 1` 并回调（226-228 行）。`process_image_task` 用 `_on_tile` 回调把 `processedFrames`/`totalFrames` 写入任务状态（图片模式下这两个字段代表 Tile 数）。CPU 模拟模式（单次 resize）不触发回调。
- **app.js**：进度日志图片模式显示 `Tile 3/10, processing, 45%`；视频模式仍显示 `3/10 帧`。进度面板图片模式 `3 / 10 Tile`，视频 `3 / 10 帧`。判断依据 `task.type === 'image'`。

## 第三轮：图片进度条按 Tile 百分比驱动 + 初始日志 0/Tiles（commit 138411c）

### 问题（用户反馈）
1. 进度条一直卡在 30% —— `process_image_task` 写死 `progress=30`，tile 回调只更新 processedFrames/totalFrames 没更新 progress
2. 首条日志显示 `0/?` —— 初始 totalFrames=0，前端 `totalFrames || '?'` 把合法 0 当未知；且处理开始前未预置总 Tile 数

### 改动
- **server.py**：提取 `_tile_positions` 为模块级函数（`TILE_OVERLAP=32` 常量），upsample 与任务初始化共用，保证总 Tile 数一致
- 处理前预计算 `total_tiles` 并写入任务（progress=0, totalFrames=total_tiles）→ 首条日志即 `0/12`
- `_on_tile` 回调：`progress = round(processed/total*100)` 实时驱动百分比
- 移除硬编码 progress=20/30/80/90（避免回退）；postprocessing 阶段不再覆盖百分比
- CPU 模拟模式（无回调）：设为 1/1、progress=100
- **app.js**：前端显示改 `undefined/null` 判断，避免 0 被当 `?`
- **缓存版本号** v27→v28

### 验证

## 第四轮：超分设置持久化到本地文件（commit a33654e）

### 需求
超分设置修改后存到本地文件，每次打开页面都读取该文件恢复。

### 实现
浏览器 WebUI 的"本地文件"即 localStorage（用户浏览器内持久存储，跨刷新/重开生效）。
- 新增 `loadSettings()`：从 `localStorage['videosr_settings_v1']` 读取 JSON，合并覆盖
  `AppState.settings`（仅合并已存在字段，兼容新增字段；try/catch 容错）
- 新增 `saveSettings()`：将 `AppState.settings` 序列化为 JSON 写入 localStorage
- `initSettings()` 重构：先 `loadSettings()` → 再按已保存值恢复 UI
  （放大倍数按钮 active、最大尺寸宽/高输入值、keepFps/keepAudio 勾选、bitrate 值），
  每个控件的 change/input 事件都调用 `saveSettings()` 即时持久化

### 注意
- 设置持久化是纯前端行为，服务端不受影响（server 仍按请求接收 settings）
- 模式(image/video)未持久化，仅持久化超分设置面板

缓存版本号 v28→v29。
