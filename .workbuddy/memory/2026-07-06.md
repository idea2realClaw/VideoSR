## 第七轮：修复视频超分广播错误（commit 50741b9）
- 报错：`operands could not be broadcast together with shapes (1920,2560) (960,1280)`（进度 11% failed）。
- 根因：模型 real_esrgan_x4plus 固定输出 ×4（upsample 内部硬编码 SCALE=4），但 `process_video_task` 之前用前端传来的 `scale` 设置（本次=2）去算 `dst_w/dst_h` 和 `motion_compensated_interpolation(scale=scale)` 的光流上采样倍数。结果：SR 帧是 ×4(1920x2560)、光流只 ×2(960x1280) → x_coords(SR尺寸) 与 flow(SR/2) 广播炸；同时 VideoWriter 按 ×2 建、与 ×4 帧尺寸不匹配。
- 修复（只动视频相关代码，图片代码不动）：
  - 视频超分倍数恒等于模型倍率 `SCALE`（删除 `scale = settings.get('scale',4)`，dst 尺寸与插值调用统一用 `SCALE`；输出文件名也用 SCALE）。
  - 保留 max-resolution 限制（max 1920x1080）：新增 `_fit_to_output(frame, dst_w, dst_h)` 在 `out.write` 前把 ×4 帧缩放到编码器目标尺寸（尺寸已匹配则不缩放，INTER_AREA）。
  - 所有 `out.write(...)` 包裹 `_fit_to_output`。
- 验证：用 stub flask 导入 server，构造 480x640 源（×4=1920x2560、×2=960x1280）；确认 scale=2 复现广播错误、scale=SCALE 返回 (1920,2560,3) 且值域 0~255、_fit_to_output 缩放与「已匹配不缩放」路径均正确。已删测试文件。
- 注意：源 480x640 经 ×4(1920x2560) 后，max-resolution 限制会把它压到约 810x1080 写入——这是既有安全逻辑，非本次引入；若想保留完整 ×4 可在后续去掉该限制。

## 第三轮修复：超分后端分块逻辑 bug（commit 0c73011）

### 问题
- 用户测试大图超分报错：`operands could not be broadcast together with shapes (512,616,3) (2048,2048,3) (512,616,3)`
- 根因 1：`_tile_positions()` 在单边尺寸 < 512 时返回负值（如 `dim - 512`），导致 `out_x = x*SCALE` 为负，numpy 负索引从末尾倒取，切片尺寸与 2048 块不匹配
- 根因 2：`result_weight` 定义为 `(H,W,1)`，但累加 `(H,W,3)` 的 `weight_3d`，原地 `+=` 无法广播（用户只看到根因1，因为根因2在其后执行）

### 修复
- `_tile_positions()`：`if dim <= tile: return [0]`（单边<512只取一块，从0开始，PIL crop 自动补黑边）
- 裁剪起始坐标：`x_start = max(0, min(x, orig_w - tile_size))`（y 同理），保证非负
- `result_weight` 改为 `(H,W,3)`，归一化去掉 `np.repeat`，直接 `result_accum / result_weight`
- 用 mock NPU（bicubic）覆盖 12 种尺寸（含 <512、1920x1080、513x513 等）验证：无广播错误、输出尺寸正确、数值有界

### 验证环境注意
- managed Python 缺 cv2 / flask_cors；测试时用系统 Python（`C:\Users\zhuxi\AppData\Local\Programs\Python\Python312\python.exe`）+ flask stub 注入
- 真实运行环境（含 qai_appbuilder）才有 NPU，本地只能验证逻辑

## 第二轮修复：精确对齐（commit e8170db）

### 问题根因
`object-fit: contain` 由浏览器自动计算居中留白，对不同像素尺寸的图片（如原图154×128 vs 结果图616×512）可能产生**亚像素舍入差异**，导致两张图的画面内容偏移不同步。

### 修复方案：彻底干掉 object-fit: contain
新增 `fitImageToContainer()` 函数（app.js 第1433行），用 JS 精确计算：
1. `scale = min(containerWidth / naturalWidth, containerHeight / naturalHeight)`
2. `displayWidth = naturalWidth * scale`
3. `displayHeight = naturalHeight * scale`
4. `marginLeft = (containerWidth - displayWidth) / 2`
5. `marginTop = (containerHeight - displayHeight) / 2`

原图和结果图使用**完全相同的公式**，确保像素级对齐。

### 修改的文件
- **style.css**: 移除 img 的 `object-fit: contain; width:100%; height:100%`，改为由 JS 控制
- **app.js**: 新增 `fitImageToContainer()` 函数；原图和结果图加载时都调用此函数

## 第八轮：视频进度日志去重 + 速度诊断计时（commit a94e419）
- 用户实测 79帧 640×480 视频耗时 132 秒，前端日志进度大量重复（50%→80%→95% 阶梯跳跃、相同值反复刷屏）。
- 进度不连续根因：
  - 后端 `process_video_task` 进度更新被 `if frame_idx % 10 == 0` 节流，只更新约 11 次 → 阶梯跳跃；
  - 前端 `startProgressPolling` 每次轮询**无条件** `Logger.info('进度更新...')` → 相同值反复刷屏；
  - `frame_idx` 计数 bug：写偶数帧时 `frame_idx += 2`（实际只写 1 帧），导致 frame_idx 最终到 118 而 total_frames=79，进度分母失真（被 min(progress,95) 掩盖）。
- 修复（只动视频路径与前端，不动图片代码、不动 _infer_tile）：
  - 后端：进度更新去掉 %10 节流（每轮更新一次，平滑显示）；`frame_idx += 2` → `frame_idx += 1`；progress 公式不变（修复后 frame_idx=实际写出帧数，最大=79 不再超 100）。
  - 前端：轮询仅在 `status`/`progress` 真正变化时才 `Logger.info`；`startProgressPolling` 开头重置 `_lastLogStatus/_lastLogProgress` 去重标记。
  - 前端缓存版本号 ?v=2026070620 → ?v=2026070621。
- 速度诊断（关键）：132s ÷ 80 次 NPU 推理 ≈ 1.65s/次。512→2048 的 ESRGAN x4plus 在 HTP 上不可能这么慢，**几乎可确定 NPU 没真正加速（回退 CPU 或 QNN Inference 开销巨大）**。
  - 在 `process_video_task` 内对 `upsample()`、`motion_compensated_interpolation()` 分别 `time.time()` 计时累加，结尾 print：NPU(upsample) 总耗时/平均单次、光流合成总耗时、其他(IO/resize/融合)。**未改 _infer_tile，避免动到图片代码**。
  - 待用户重跑看日志 `平均: X.XXX 秒/次` 确认瓶颈；若 NPU 平均 1s+ 则需排查 HTP 是否真启用（QNNContext 是否静默回退 CPU）。
